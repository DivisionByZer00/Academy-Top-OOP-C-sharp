## Homework 01

### Тема: "От классов к архитектуре. Мышление объектами"

***Задание:*** Вам необходимо создать простую систему для учета товаров на небольшом складе (инвентарь). Программа не будет иметь интерактивного меню, ее задача — продемонстрировать правильное проектирование моделей данных и класса, который ими управляет.

***Логика работы программы***
- **Модель Product:** Вы должны спроектировать и реализовать record для представления товара. Он должен хранить уникальный артикул (ID), название и цену.
- **Класс Inventory:** Вам нужно создать класс, который будет управлять списком товаров. Этот класс должен инкапсулировать (скрывать) в себе коллекцию List<Product> и предоставлять публичные методы для работы с ней.
- **Демонстрация в Main:** В основном методе программы вы должны продемонстрировать работу вашей системы: создать инвентарь, добавить в него несколько разных товаров, а затем найти один из товаров по его артикулу и вывести информацию о нем.

## Homework 02

### Тема: "Наследование. Построение иерархий и переиспользование кода"

***Задание:*** Спроектировать и реализовать простую модель для расчета заработной платы разных типов сотрудников в компании. Необходимо построить иерархию классов, где Manager и Developer являются наследниками общего класса Employee, и продемонстрировать полиморфную работу с
ними для расчета и вывода зарплаты каждого сотрудника.

***Логика работы программы***
- **Базовый класс Employee:** необходимо реализовать базовый класс для всех сотрудников. Он должен содержать общие свойства Name и BaseSalary, а также виртуальный метод CalculateMonthlySalary(), который по умолчанию возвращает базовую зарплату.
- **Производный класс Manager:** необходимо реализовать класс, наследуемый от Employee. Он должен иметь дополнительное свойство Bonus и переопределять метод CalculateMonthlySalary() для учета этого бонуса.
- **Производный класс Developer:** необходимо реализовать класс, наследуемый от Employee. Он должен иметь дополнительное свойство LinesOfCodeWritten и переопределять метод CalculateMonthlySalary() для расчета премии на основе написанного кода.
- **Демонстрация в Main:** в основном методе программы вы должны продемонстрировать полиморфную работу системы: создать список List<Employee>, добавить в него объекты Manager и Developer, а затем в цикле вывести зарплату для каждого, вызывая их индивидуальные версии метода CalculateMonthlySalary().

## Homework 03

### Тема: "Полиморфизм и Абстракция. Создание гибких систем"

***Задание:*** Спроектировать и реализовать систему для рендеринга (т.е. отображения) различных типов документов. Необходимо создать абстрактный базовый класс Document и несколько производных классов для конкретных типов документов (TextDocument, ImageDocument). Программа должна продемонстрировать полиморфную обработку коллекции, содержащей разные типы документов.

***Логика работы программы***
- **Абстрактный класс Document:** необходимо реализовать абстрактный базовый класс. Он должен содержать общее свойство, например Author (string), и абстрактный метод Render(), который не имеет реализации.
- **Производный класс TextDocument:** необходимо реализовать класс, наследуемый от Document. Он должен иметь дополнительное свойство Content (string) и переопределять (override) метод Render() так, чтобы он выводил в консоль текстовое содержимое документа.
- **Производный класс ImageDocument:** необходимо реализовать класс, наследуемый от Document. Он должен иметь дополнительное свойство Resolution (string, например "1920x1080") и переопределять метод Render() так, чтобы он выводил сообщение о рендеринге изображения с указанием его разрешения.
- **Демонстрация в Main:** в основном методе программы вы должны продемонстрировать полиморфную работу системы: создать список List<Document>, добавить в него объекты TextDocument и ImageDocument, а затем в цикле для каждого документа вызвать метод Render(), чтобы увидеть его уникальную реализацию.

## Homework 04

### Тема: "static и паттерн Одиночка. Управление состоянием на уровне класса"

***Задание:*** Спроектировать и реализовать простой сервис для кэширования данных в приложении. Кэш — это хранилище, которое позволяет быстро получать доступ к часто используемым данным, не загружая их каждый раз из медленного источника (например, из файла или базы данных). В приложении должен существовать только один экземпляр кэша, доступный из любой его части. Для решения этой задачи необходимо использовать паттерн Одиночка.

***Логика работы программы***
- **Класс CacheService:** необходимо реализовать запечатанный (sealed) класс, который будет управлять кэшем. Этот класс должен быть реализован по паттерну "Одиночка", хранить внутри private Dictionary<string, object> и предоставлять публичные методы void Add(string key, object value) и object? Get(string key).
- **Демонстрация в Main:** в основном методе программы вы должны продемонстрировать, что кэш является глобальным и единственным: получить экземпляр сервиса дважды в разные переменные, добавить данные через одну, а прочитать через другую, и в конце доказать с помощью object.ReferenceEquals(), что обе переменные ссылаются на один и тот же объект.

## Homework 05

### Тема: "Интерфейсы. Создание гибких контрактов для классов"

***Задание:*** Вам предстоит создать небольшую консольную программу для обработки текста, архитектура которой будет построена на "плагинах". Каждый плагин — это отдельный класс, выполняющий одно конкретное преобразование текста (например, перевод в верхний регистр или удаление
пробелов).    
Главный класс-обработчик не должен ничего знать о конкретных плагинах. Он должен уметь работать с любым объектом, который реализует общий интерфейс ITextPlugin, что позволит в будущем легко добавлять новые способы обработки текста, не меняя основной логики.

***Логика работы программы***
- **Интерфейс ITextPlugin:** вы должны спроектировать и реализовать интерфейс, который будет служить контрактом для всех плагинов. Он должен определять один метод, например, string Process(string input), который принимает на вход строку и возвращает измененную строку.
- **Классы-плагины:** Вам нужно создать как минимум три разных класса, реализующих интерфейс ITextPlugin:
  - **ToUpperPlugin:** преобразует всю строку в верхний регистр.
  - **SpaceRemoverPlugin:** удаляет все пробелы из строки.
  - **ReversePlugin:** переворачивает строку задом наперед.
- **Класс TextProcessor:** создайте класс, который будет управлять процессом обработки. У него должен быть метод, который принимает на вход исходную строку и список (List<ITextPlugin>) плагинов. Этот метод должен последовательно применять каждый плагин из списка к строке и возвращать итоговый результат.

## Homework 06

### Тема: "Делегаты, События и Паттерн Наблюдатель"

***Задание:*** Вам предстоит создать консольную симуляцию простого аукциона. В системе будет один главный объект — AuctionLot (аукционный лот), который будет "издателем". Также будет несколько объектов-участников Bidder, которые будут "подписчиками".    
Когда новый участник делает ставку на лот, AuctionLot должен уведомить всех остальных участников (кроме того, кто сделал ставку) о том, что цена изменилась. Это классический пример паттерна "Наблюдатель", который вы должны реализовать с помощью механизма событий C#.

***Логика работы программы***
1. **Класс Bidder (Участник):** Вам нужно создать класс, представляющий участника аукциона. У него должно быть имя и метод, который будет реагировать на изменение цены (например, OnPriceChanged). Этот метод будет обработчиком события.
2. **Класс AuctionLot (Лот):** Это главный класс-издатель.
- Он должен содержать информацию о лоте (например, название и текущую цену).
- Он должен объявлять событие (event), на которое смогут подписываться участники. Событие должно передавать информацию о новой цене и о том, кто ее предложил.
- У него должен быть публичный метод PlaceBid(Bidder bidder, decimal newPrice), который позволяет участнику сделать ставку. Внутри этого метода должна быть логика проверки (новая ставка должна быть выше текущей) и генерация события для уведомления остальных.
3. **Демонстрация в Main:** В основном методе программы вы должны:
- Создать один экземпляр AuctionLot.
- Создать несколько экземпляров Bidder.
- Подписать всех участников на событие лота.
- Сымитировать торги, вызывая метод PlaceBid от имени разных участников и наблюдая, как остальные получают уведомления.

## Homework 07

### Тема: "S - Принцип единственной ответственности"

***Задание:*** Вам будет предоставлен один плохой класс OrderProcessor, который нарушает Принцип единственной ответственности. Он занимается всем сразу: получает данные о заказе из консоли, проверяет их (валидирует), сохраняет в файл и выводит уведомления.    
Ваша задача — выполнить рефакторинг этого класса, разделив его на несколько маленьких, сфокусированных классов-"специалистов". Каждый новый класс должен иметь только одну, четко определенную обязанность.

***Логика работы программы***
- **Исходный код:** Вы должны начать с предоставленного ниже "класса-бога". Ваша задача — не изменить его внешнее поведение, а улучшить внутреннюю структуру.    
*// Этот класс нужно отрефакторить    
public class OrderProcessor    
  {    
  public void ProcessOrder(string itemName, int quantity)    
    {    
    // 1. Валидация    
    if (string.IsNullOrEmpty(itemName))    
    {    
      Console.WriteLine("Ошибка: Название товара не может быть пустым.");    
      return;    
    }    
    if (quantity <= 0)    
    {    
      Console.WriteLine("Ошибка: Количество должно быть больше нуля.");    
      return;    
    }    
    Console.WriteLine("Заказ прошел валидацию.");    
    // 2. Сохранение    
    File.WriteAllText("order.txt", $"Товар: {itemName}, Количество: {quantity}");    
    Console.WriteLine("Заказ сохранен в файл.");    
    // 3. Уведомление
    Console.WriteLine("Отправка email-уведомления: 'Ваш заказ принят'.");    
  }    
}*

**Классы-специалисты:** Вы должны создать как минимум три новых класса:
- OrderValidator: Должен отвечать только за проверку корректности данных заказа.
- OrderRepository: Должен отвечать только за сохранение заказа (например, в файл).
- NotificationService: Должен отвечать только за отправку уведомлений пользователю.

**Класс-оркестратор:** Исходный класс OrderProcessor (или новый) должен стать дирижером. Его единственной задачей будет координация работы валидатора, репозитория и сервиса ууведомлений.



## Homework 09

### Тема: "L - Принцип подстановки Барбары Лисков. Паттерн Шаблонный Метод"

***Задание:*** Вам предстоит создать небольшую консольную программу для обработки и анализа данных из различных источников (например, CSV и XML). Архитектура программы должна быть построена с использованием паттерна Шаблонный методи.    
Вы создадите абстрактный базовый класс, который будет определять общий "скелет" алгоритма обработки данных. Конкретные классы-наследники будут реализовывать специфичные шаги этого алгоритма (например, чтение и парсинг), не изменяя при этом его общую структуру.

***Логика работы программы***
- **Абстрактный класс DataProcessor:** Вам нужно спроектировать и реализовать абстрактный класс, который будет служить основой для всех обработчиков.
  - Он должен содержать public шаблонный метод Process(), который определяет последовательность операций: ReadData(), ParseData(), AnalyzeData(), SaveReport().
  - Методы ReadData() и ParseData() должны быть protected abstract, так как они будут уникальны для каждого источника данных.
  - Методы AnalyzeData() и SaveReport() должны быть protected virtual и иметь реализацию по умолчанию (например, AnalyzeData просто считает количество записей, а SaveReport выводит результат в консоль).
- **Классы-наследники:** Вам нужно создать как минимум два разных класса, наследующихся от DataProcessor:
  - CsvDataProcessor: Реализует ReadData для "чтения" данных из строки в формате CSV. Реализует ParseData для разбора этой строки.
  - XmlDataProcessor: Реализует ReadData для "чтения" данных из строки в формате XML. Реализует ParseData для разбора этой строки.
- **Демонстрация в Main:** В основном методе программы вы должны продемонстрировать полиморфную работу вашей системы: создать объекты CsvDataProcessor и XmlDataProcessor, а затем вызвать для каждого из них один и тот же шаблонный метод Process(), чтобы показать, как он работает для разных реализаций.
